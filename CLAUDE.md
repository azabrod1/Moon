# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Development

```bash
npm run dev        # Start Vite dev server (hot reload)
npm run build      # TypeScript check + Vite production build (output: dist/)
npm test           # vitest run — astronomy golden/consistency tests (src/astronomy/*.test.ts)
npm run preview    # Serve production build locally
npm run gen:stars  # Regenerate the bright-star catalog (gen-stars.mjs)
npm run gen:moons  # Regenerate satellite elements + goldens from JPL (gen-moons.mjs; --offline uses .moon-data-cache/)
npm run gen:constellations  # Regenerate constellation figures from HYG+Stellarium (gen-constellations.mjs → src/planetarium/data/constellations.ts)
npm run gen:maps   # Regenerate derived texture maps (gen-maps.mjs; ocean-glint roughness, height→normal — runs in headless Chromium)
```

Dev bridge (manual testing): in `npm run dev`, the `window.__moon` console bridge drives the app so you never click there by hand (DEV-only; dead-code-eliminated from production). Camera/clock: `jumpTo('Pluto')`, `frame('Pluto', fill, phaseDeg)`, `setTimeMs(Date.parse('2026-07-14T00:00:00Z'))`, `setTimeRate(21600)`, `setTimePaused(true)`, `setFov(deg)`, `setChrome(false)`, `bodies()`, `probe(name)`. The landed/Observatory pipeline has real hooks — `land('Io')`, `openObservatory()`, `lookUp()`, `lookAt('Europa')`, `swapVantage()`, `jumpEvent('lunar-eclipse', 1)`, `exitSurface()`, `probeLanded()`; `frame()` only poses the camera and never routes through the landed code, so exercise landed-mode changes through these. The tutorial drives through `tutorialStart()`, `tutorialNext()`, `tutorialBack()`, `tutorialSkip()`, `tutorialState()` (one flat snapshot: step/phase/arrival/clock — poll `phase === 'ready'` between stops). Motion forensics: `traceStart('Moon')` / `traceStop()` record one render-truth sample per cruise frame (screen-space body position, frame pacing, sim clock, camera, speeds) into a preallocated buffer — the tool that separates position glitches from pacing hitches when chasing stutter.

Headless screenshots: `node tools/shoot.mjs` drives the same bridge (`--label`, `--bodies`, `--time`, `--fill`, `--phase`). It — and any ad-hoc Playwright capture script — must launch Chromium with the real-GPU flags `--use-gl=angle --use-angle=metal --enable-gpu --ignore-gpu-blocklist`: the GPU/ANGLE (Metal) path renders correct frames ~15× faster, while the SwiftShader software path can return black frames (`--software` is a fallback only). The core visual check is **before/after** — two screenshots of the identical view (same body/time/fill/phase) to see what a change helped or hurt. Full reference (method details, the before/after recipe, the GPU gotchas) is in the **`headless-webgl-screenshots`** skill. `tools/` also holds the wider capture/forensics kit — `moon-shot.mjs`, `compare.mjs`, `montage.mjs` (framing + composites), `diffcrop.mjs`, `colormatch.mjs`, `chstat.mjs` (pixel diffs / colour match / channel stats), `surface-sweep.mjs` (numeric probe sweeps), `gpu-verify.mjs` (MoonPainter GPU-paint check) — look there before writing a new one. Ad-hoc Playwright scripts must live inside the repo (`tools/` or `planning/`), never `/tmp`: ESM resolves `node_modules` from the script's location.

No linter is configured. **Run `npm run build` and `npm test` locally after every change.** CI (`.github/workflows/deploy.yml`) runs `tsc`, the tests, then the Pages build. `tsconfig` is `strict` and sets `noUnusedLocals`/`noUnusedParameters`, so unused imports, locals, and parameters fail the build (this is what catches refactor leftovers). Deploys ship by pushing `main` (feature branches fast-forward in); the Pages upload step flakes, so `gh run rerun <id> --failed` once or twice before diagnosing a red deploy. `planning/` is gitignored local scratch (design docs, mockups, QA scripts) — never commit it; stage by explicit path rather than `git add -A`.

Tests are colocated `*.test.ts` next to their modules (vitest, explicit imports, no config file) — `src/astronomy/*` plus a growing set under `src/planetarium/` (`SolarSystem`, `deckLogic`, `arrivalLogic`, `orbitDetails`, `surfaceView`, `landedView`, `observatoryTime`, `PlanetariumStore`, `ui/ObservatoryPanel`, and `world/{texturePolicy,surfaceShading,planetshine,MoonPainter}`). They pin the ephemeris to Meeus worked examples, published event catalogs (full moons, eclipses), the scene's frame convention, and JPL Horizons vector goldens: 45 heliocentric (`standish.test.ts` — provenance in its header) pinning the Standish propagation and the J2000 frame absolutely, ~800 planetocentric moon vectors (`satellites.goldens.json`, bounds derived from each record's measured residuals), and shadow-event goldens (`shadows.goldens.json` — Horizons vectors at eclipse/transit peaks + EclipseWise Earth-eclipse circumstances; regeneration recipe in the `shadows.test.ts` header). When astronomy math changes, update fixtures deliberately, never by copying the new output; the moon fixtures regenerate only via `npm run gen:moons`.

## Architecture

A **Three.js** WebGL app: the **Planetarium** is the app, plus a dormant **Moon Flight** mini-game sharing the scene, renderer, and animation loop. `src/main.ts` is a thin entry point: it owns the renderer, the post-processing composer, the two cameras, mode switching/transitions, and the animation loop. Each mode is a self-contained controller that exposes `activate()/deactivate()/update(dt)`. (The legacy "Moon view" mode retired in favor of the Observatory — recoverable from git history; its orbit-details overlay was ported first.)

### Planetarium — `src/planetarium/`
A fly-through of the full solar system (Mercury–Pluto) in **AU**. Uses a **floating-origin** pattern: the player stays at scene origin and all bodies are offset by the player's position, avoiding float precision loss at large distances.
- **`PlanetariumMode.ts`** — the mode controller. Self-contained pieces are extracted to siblings (`world/starfield`, `input/GyroSteering`, `ui/*` panels, labels); the tightly-coupled per-frame core (update pipeline, navigation, landing, missions, camera) deliberately stays in the controller. **`cruiseView.ts`** is the cruise rig's pure module: every chase-rig distance (hull reference radius, chase trail, clearance pads, camera margin, zoom floor) derives from ONE scaled base — currently 1/32 of the legacy Moon-scale rig, parking max approach at the 4K texture's acceptable edge — plus the camera-safety math (`escapeCameraPenetrations`: one ray step provably clears ALL padded body shells at once; conjunct co-orbital moons' padded shells can still overlap) and the dynamic cruise near plane (30% of the tightest of body surface / camera-to-hull / ring annulus, clamped [3 km, 135 km] — rings have no collision, so the near term is what stops a ring skim clipping a hole). The per-frame safety pass runs AFTER `updateMoonPositions` (at 1 yr/s a capped frame moves moons 36 simulated days) and is skipped under `devFreeCamera`; the landed near is `landedNearAU(trueRadius)` (`landedView.ts` — the stock `LANDED_NEAR_AU`, shrunk on bodies smaller than it so surface view still culls the ground and tiny moons keep the full frame), applied in `applyLandedTarget` before the framing math reads `camera.near`; deactivation restores the stock value. **`moonRenderSize.ts`** is the moon rendered-size policy: below an anchor fraction of the parent's radius (5% flythrough, 2.5% observing the planet, 0 = true size in surface view), moons inflate toward the anchor on a compressive curve (`anchor·(true/anchor)^γ`, γ 0.4) — size ordering survives instead of every small moon pinning to one marble; all knobs live there, and the dev bridge's `setMoonSizeGamma(γ)` retunes live.
- **`SolarSystem.ts`**, **`PlanetFactory.ts`** — build planets, orbit lines, asteroid belt, rings; async texture loading with procedural canvas fallback.
- **`PlayerShip.ts`** — player state + kinematics; the procedural spacecraft geometry lives in **`ship/models/`** (`shipPrimitives`, `defaultShip`, `voyager`, `cassini`, `newHorizons`, `juno`).
- **`PlanetLabels.ts`**, **`Constellations.ts`**, **`ui/SunLabel.ts`** — billboard sprites + HTML labels with foreground-disc occlusion culling.
- **`PlanetariumStore.ts`** — localStorage/sessionStorage/IndexedDB persistence with 30s auto-save; migrates the pre-rename `orbital-sim-explore-state` key and reads the legacy `explore-help-seen` flag (intentional back-compat). `skyPref` is deliberately tri-state: absent from the save until the user flips the toggle (the device default — on for fine pointers — resolves at read, so the auto-save can't bake it in).
- **The deck** (`deckLogic.ts` — pure row filter, observe-arrival decision table, catalog grouping; widgets + controller in `PlanetariumMode`): the single body picker replacing the old travel/observatory menus — three tabs (Autopilot · Travel · Observatory), one-click rows under sticky system headers with a HERE pill on the landed row, one shared search whose query survives tab switches (moons match on their own or parent name, a planet stays when one of its moons hits, **no sibling ride-along**), and a ≤640px bottom-sheet form. Commits by tab: **Observatory** lands there and `observeArrivalAction` picks reopen / land-with-panel / land-quiet from `skyPref`, the panel's From ▾ exception, and companionless-always-quiet (quiet arrivals pulse the telescope chip); **Travel** jumps moon-precise via `getMoonJumpDestination` (position derived live from the parent + ephemeris offset — never `moonWorldPositions`; the pose math is `arrivalLogic.moonArrivalPose`, pure + catalog-swept: standoff targets a 5° apparent disc from the chase camera with a ~500 km floor and 45%-of-separation cap, sun-side preferred with parent/ring/sightline rejection and outward-radial fallback, and the heading aims past the limb so full thrust is a flyby, parent flanking the opposite third) and your own row lifts off. Arrivals stay under way: `arrivalLogic.governedSpeedCap` caps closing speed at K × surface distance near every governed body — visible painted moons (plus the jump target seeded through the veil window), planets at their atmosphere-ENVELOPE radii (`ATMOSPHERE_SHELL_SCALES` — Jupiter/Saturn's 1.015R shells are the outermost surface), and the Sun at 1.2 photosphere radii (no collision shell; the glide is the only brake); the override latch is `advanceBodyCap` (binds-vs-COMMANDED speed + 0.75 s sustained clear-hold, whole state reset on every flight discontinuity). Moon meshes get segment-swept collision pushback, and throttle-up revives a parked ship — teleports never park (the tutorial's Saturn freeze-frame parks explicitly; `frame()`/dev jumps park for capture); **Autopilot** engages the cruise. Keys: `T`/`O`/`P` everywhere outside missions/help (landed and surface view included); Esc cascade help → deck → stats → Look-at → surface view → panel → takeoff; while open, printable keys focus the search — keys can open the deck but never close or switch it.
- **The time rail** (`ui/PlanetariumTimePanel.ts` — DOM-thin widget + pure rail math `railFraction`/`nearestPresetIndex`/`railDetentLabel`, colocated test; presets in `timeRates.ts` — the widget derives detent count/labels/positions from the array, never hardcodes nine): the bottom bar's time control — a magnitude-only rate rail riding on an adaptive clock (`formatAdaptiveClock` coarsens the readout as the rate climbs; zero-padded day for stable width), with the expanded panel (clock tap) owning reverse, the full labeled rail, a minute-precision `datetime-local` (deliberate deviation from the handoff's read-only seconds readout — typing a date must keep working; change → rebuild positions + restart the event search), and Now. Drag/tap snaps to a detent **preserving the rate's sign**; tap-in-place pauses (pixel-slop, not detent-based); off-ladder rates (tutorial's 2 hr/s) sit log-linearly between detents and label honestly through `formatTimeRateLabel` (wk/mo tiers; Julian-year divisor so the top detent reads "1 yr/s"). Ember = live at now (|rate| exactly 1, forward, within 120 s), accent = warped, hollow/grey = paused; the unit flash over the clock is **gesture-gated** (`updateTimeUI({flash})`) so restores never flash. Keys: `,` `.` step, `N` now (t/o/p guard set + ☰-menu lock — the menu auto-pauses the clock, so gestures and keys both go inert while it's open); Space pauses **only landed** (cruise keeps ship thrust); the rail/clock handle their own keys and stop propagation, and the window handler ignores `defaultPrevented` and Space-on-buttons. The panel is `.visible`-class, closes on click outside `#time-control` (Space cluster counts as outside), pairs one-instrument-at-a-time with Stats (not the Observatory panel — deliberate popover parity), and joins the Esc cascade + every lifecycle close path; landing keeps the rail interactive (only the speed group goes inert). The widget's `bind()` re-caches elements every activation but wires listeners once behind a `wired` flag.
- **The tutorial** (`tutorialLogic.ts` — pure step deck with copy as data, phase machine, settle predicate, restore-sequencing table; `ui/TutorialCard.ts` — DOM-thin narrator card over a pure `tutorialCardModel`; card copy quotes the action buttons via `{teleport}`/`{observatory}` tokens rendered as clones of the live cluster SVGs; driver as `PlanetariumMode` members): six Next-driven cards staging showcase scenes — deck theater into Saturn (Teleport tab, row pulse, then the same commit a click makes), Observatory-tab theater onto the Moon with the panel opened regardless of `skyPref`, the Jupiter system at 2 hr/s (deliberately off the preset ladder — every readout formats the live rate) from the landed orbit camera pulled to 8 radii with the Observatory panel kept open (planet spins, Galileans wheel, the panel's now-bar and phase run visibly), and the pinned 2027-08-02 eclipse (searched from a fixed date — goldens-pinned in `shadows.test.ts`) at 20 min/s from first contact, self-settling to 1× just inside totality on the spot-riding surface camera with the Observatory panel reopened over the sky (the card teaches Observatory → Earth → Surface view → next Solar Eclipse as the reproducible path, so the panel must be visible; the eclipse card's Next stays held until totality). Every arrival routes through a generation-checked idle-commit wrapper (`arriveThen` drops rival calls; the tutorial retries until idle — the tutorial wins, a beat later). While active, `getState()` serves the pre-tutorial snapshot, so autosave/Save/pagehide/deactivate all keep the journey the user left and a mid-tutorial reload resumes it; every exit (Esc — its cascade rung sits above the deck's — skip, End tutorial, deactivate, mission start) restores position, landing, and clock; Back (cards 3+) re-stages the previous stop through the same absolute stagings; banners mute while the card narrates (`show(msg, {force})` pierces for Save + end toasts). Entries: help-modal action bar (primary styling on the first-run showing only; revisits get a plain button and no "Explore on my own") + ☰ "Tutorial" (disabled during missions and while running). **`data/`** (`brightStars`, `constellations`), **`world/`** (`starfield`; `texturePolicy` — the single texture-creation policy: device-cap capture (anisotropy, max size), colour-space by map kind, and 2K/4K tier→URL resolution; `surfaceShading` — per-body `onBeforeCompile` injection (directional starlight night-floor, per-archetype `LIMB_DARKENING`, Saturn ring shadow, moon-transit umbra spots); `planetshine` — earthshine night-side lift (pure + tested); `MoonPainter` — lazy, frame-budgeted procedural moon textures behind the never-show-unpainted gate; `ShadowVisuals` — true-scale shadow cones + transit shadow spots for the landed system; `OrbitDetailsVisuals` — orbit-details overlay geometry; `deprecatedSkybox` — unwired, kept for reference), **`input/`** (`GyroSteering`; `SurfaceLook` — surface-view drag-look + wheel/pinch FOV), **`ship/models/`**, **`ui/`** (panel classes with `bind()`/`render()`).
- **Observatory** (`ui/ObservatoryPanel.ts` + `ui/ObservatoryHUD.ts` + `surfaceView.ts`): the landed-mode panel (Option D "Cinematic Instrument" — "From ⟨body⟩ ▾" vantage button + companion swap, SVG phase glyph with a mono ∅/distance line, live now-bar, surface-view entry). The vantage button opens the deck's Observatory tab with the panel kept open behind; that pick's arrival keeps the panel even on a quiet preference. **Every planet is a subject**: moonless systems (Mercury, Venus) render the companionless "Quiet sky" variant — catalog-tinted ring glyph, no swap chip/events/magnitudes/orbit footer, empty event search — and always arrive quiet. Earth keeps its four prev/next rows with next-date metas (eclipse pairs run on the shadow engine and reuse the upcoming-search results; full/new moon stay on `findEvent`); every system gets an upcoming-events list searched chunked at ~4 ms/frame via `searchShadowEvent` resume cursors — restarted on open/jump/date-set, never steady-state. Magnitudes display for Earth only (generic systems badge-only). Eclipsed moons dim per frame via `computeMoonShading` in `updateMoonPositions` (Earth's Moon reddens in umbra). **Orbit details** (footer toggle, ported from Moon view): `orbitDetails.ts` (pure, unit-tested) derives ellipse annotation from the moon's REAL trajectory sampled through `computeMoonOffsetEquatorialAU` — apsides = radial extremes, center/major axis = the curve's max-diameter pair (apsides-midpoint skews on the perturbed Meeus Moon); `world/OrbitDetailsVisuals.ts` draws line/axes/Kepler sectors in the system group; F1/F2 foci are fixed-px HTML glyphs (sub-pixel as geometry for irregulars). Subject = the moon of the vantage pair (sticky across moon→parent swaps via `orbitPairMoon`); periods come from `getMoonDisplayOrbit` (mean-longitude rate — `360/|Ṁ|` is 30× wrong for the degenerate Tethys-family fits); landed `maxDistance` stretches to the subject's apoapsis while the toggle is on. Session-only, hidden in surface view.
- **Surface view** ("Look up"): a narrow-FOV (1.5–45°) look-from-the-surface camera, session-only landed sub-state. `surfaceView.ts` is the pure observer-circumstances table — per-event targets incl. standing in the umbral spot for solar-eclipse views (`shadowAxisSurfacePoint` in shadows.ts, shared with `ShadowVisuals.poseSpot`) — plus vantage/entry-FOV math, unit-tested in `surfaceView.test.ts`. While active: OrbitControls hand over to `input/SurfaceLook`, the landed system's moons drop the render-curve inflation (angular sizes are real — Io's silhouette is Io-sized), planet/moon/Sun labels and orbit lines hide, and the camera re-pins at the end of `updateLanded` (after the position refreshes). Event jumps re-point an active surface view instead of orbit-framing; the vantage swap re-lands on the companion via `applyLandedTarget` (no exit/enter ceremony, `preLand*` untouched). **Look-at picker**: on a generic planet with moons and no live event, Look up opens a target menu (`ui/SurfaceTargetMenu.ts`, a thin DOM card; choice building/ordering/keying are pure functions in `surfaceView.ts`) instead of defaulting to the Sun; a ⌖ chip in the surface HUD (`#surface-lookat`) re-opens it while in surface view; a pick is remembered (`surfacePickedTarget`, cleared when the ground changes or on takeoff) but a live event always outranks it. The surface HUD (`ui/ObservatoryHUD.ts`) measures the bottom bands' top edge at its 8 Hz text pass and clamps the tracking cluster above it every marker frame, so the overlays can't collide on narrow phones.

### Moon flight — `src/moonFlight/`
Lunar-landing mini-game and the **cleanest decomposition to emulate**: `MoonFlightMode` (thin controller) composes `FlightController` (physics), `FlightInput` (keyboard/touch), `FlightHUD` (DOM), `SkyScene`, `lightingSnapshot`. Note: there is currently **no UI entry point** to this mode (the button was removed in git history); the code is intact and reachable only by restoring an entry call.

### Astronomy — `src/astronomy/`
Meeus ephemeris + JPL/Standish planet elements: `ephemeris` (Sun/Moon ecliptic-of-date position, phase, event search), `standish` (Standish/JPL propagated Kepler elements — the planets' element source; Table 1 inside 1800–2050, Table 2 beyond with clamped 3000 BC–3000 AD validity; values transcribed verbatim, never re-round), `satellites` + `satelliteElements` (all 64 non-Earth moons: JPL mean-element geometry with Horizons-anchored phase and per-moon calibrated/fitted rates — `satelliteElements.ts` is **generated**, edit only via `gen-moons.mjs`; its provenance header lists per-moon measured residuals and tiers, incl. honest librator/irregular limits), `planetary` (element→scene-vector math, the Meeus Earth/Moon seams, body state), `shadows` (analytic umbra/penumbra geometry + eclipse/shadow-transit event search: kind-split classification — immersion magnitudes for moons in a parent's shadow, near-surface cone contact for a moon's shadow on the parent; eclipse-season β-prefilter with drift-bounded strides; resumable via `timeBudgetMs` cursors; positions resolve through `computeBodyPositionAU` and the `computeMoonOffsetEquatorialAU` seam in `satellites.ts`, so the engine sees exactly what the renderer draws; mirror-invariant — dots and norms only), `precession` (of-date→J2000 longitude), `deltaT` (Espenak–Meeus ΔT; theories evaluate at TT), and `constants` (`J2000`, `OBLIQUITY_DEG`, and `DEG`/`RAD` re-exported from `shared/math/angles`).

**Frame contract: right-handed J2000 everywhere.** The scene is the J2000 equatorial frame as a **proper rotation** (det +1): +X vernal equinox, +Y celestial north, +Z = RA 270°; the intermediate ecliptic frame runs longitude toward **−Z**. `raDecToVector` (planetary.ts) is the single chirality definition site — the starfield and Constellations route through it; never re-inline the formula. Planets are native-J2000 Standish; Meeus Earth/Moon longitudes (ecliptic-of-date) are precessed to J2000 at the `planetary.ts` vector seams — never inside `ephemeris.ts` (its goldens quote of-date values). Earth's render position stays Meeus (−Sun vector, precessed) for exact Sun–Earth–Moon coherence; Earth's Standish EMB row draws only the decorative orbit line. Constellation figures have real-sky chirality, and **absolute rotation phase is real**: IAU pole + node (prime reference at RA = pole RA + 90°) + verbatim W means the right continents face the Sun at a UTC instant — pinned by the GMST test and the chirality pin (`raDecToVector(0,0) × raDecToVector(90,0) ≈ +Y`) in `planetary.test.ts`. All sign conventions are deliberate and test-pinned: don't flip any in isolation. JSON goldens are stored in raw Horizons frames; only the test-side `horizonsToScene` mappings encode the scene convention (ecliptic-J2000 `(x, y, z)` → `(x, z, −y)`).

### Shared — `src/shared/`
Cross-cutting, framework-free helpers: `math/` (`angles` — the single source for `DEG2RAD`/`RAD2DEG`; `smoothstepUnclamped`), `three/projectToScreen` (world→screen projection with an optional zero-alloc `out`), `dom` (`setText`), `debug`, `assets/` (`textures` — the Moon-flight texture URLs), `constants/` (`physicalData`), `shaders/` (`atmosphere`, `sun`). **`src/app/`** holds app-shell helpers (`gpuCapability` — the float-FBO bloom probe).

### Rendering Pipeline
- Post-processing bloom (`UnrealBloomPass`) on desktop when the GPU supports float framebuffers (tested at startup via a float FBO + `FRAMEBUFFER_COMPLETE`). Mobile/incapable GPUs get a no-bloom fallback with extra Sun glow shells. The composer is rebuilt when switching cameras between modes.
- PCF shadow maps for eclipse shadows. Custom GLSL: Sun corona (fbm noise), Earth night lights, and a per-planet single-scatter atmosphere shell (`shaders/atmosphere` — Rayleigh day-limb that warms toward the terminator + a Henyey-Greenstein Mie forward-scatter crescent, additive HDR so the composer's tonemap + bloom carry the bright rim).
- Per-body surface lighting is injected into the standard material through a shared `onBeforeCompile` hook (`world/surfaceShading`): a dim directional starlight night-floor, per-archetype limb darkening, Saturn's ring shadow, and moon-transit umbra spots — the GLSL is byte-identical per body (only uniforms differ) so programs still share. `world/planetshine` adds the earthshine night-side lift.
- Textures load through `world/texturePolicy`: device-capability capture, colour-space by map kind, and a capability-gated 2K→4K tier upgrade-on-approach (4K variants live under `public/textures/4k/` with identical filenames).
- **Lens correction** (`shared/math/lensProjection` + `app/LensPass`, planetarium only): rectilinear projection stretches off-axis spheres into ovals (~17% at 30° off-axis at 60° FOV), so a full **stereographic** pass (conformal — every sphere renders as a true circle from any direction; measured off-axis disc aspect 0.997 vs rectilinear's 1.15) sits between bloom and the OutputPass. The pass is NOT bloom-gated: when the GPU can't float-render (so bloom is off), `buildComposer` still builds a composer for the planetarium on a plain **8-bit** target carrying just RenderPass → LensPass → OutputPass, so low-end / some mobile GPUs get round planets too (at the cost of mild HDR-highlight clipping). `?nofloat=1` forces that path on capable hardware for QA. The camera renders at an overscan FOV (~75° for a 60° display) so the warped frame's corners stay covered, and `projectToScreen` applies the same forward map so DOM overlays land on warped pixels. **`applyDesignFov` is the only legal `camera.fov` writer for the planetarium camera** (`camera.userData.lens.designFovDeg` holds what the frame displays; `camera.fov` holds the overscan — never compare against it, use `displayFovDeg()`), and appearance metering that converts angular size to on-screen size must use `lensDisplayHalfTan`, not `tan(camera.fov/2)`. Frustum/clearance math correctly keeps the render fov. A pinhole source can't feed a full stereographic frame past ~80° corners, so `lensEffectiveStrength` backs the strength down for very wide DESIGN fovs (dev fill poses can ask for >100°) — **every consumer reads `lens.effectiveStrength ?? lens.strength`, never the raw request**, or the seams disagree. `tools/oval-probe.mjs` measures disc ovality against the rectilinear prediction; `__moon.setLens(strength)` A/Bs it live, and `__moon.frame(name, fill, phase, distMul, offNdcX, offNdcY)` slides any body off-centre.

### Key Patterns
- **All distances are real**: the Planetarium works in AU throughout (1 scene unit = 1 AU).
- **All textures bundled locally** in `public/textures/` (not CDN), referenced via `import.meta.env.BASE_URL + 'textures/'`. Asset paths built from `BASE_URL` strings are invisible to both tsc and Vite — verify them by running the app.
- **UI is vanilla HTML** in `index.html` (no framework); each mode has its own overlay section toggled via `display`. The TS reads elements by `id` — keep the `index.html` id set and the `getElementById` calls in sync.
- **Nothing half-loaded is ever shown**: a moon never renders before `MoonPainter` finishes it, and teleports hold the arrival veil until the destination scene is ready. Perf work must preserve this gate.
- **Transient overlays are one-modal-at-a-time**: the ☰ menu, the deck, and the Look-at menu each close the others on open. Landing and takeoff close the deck and the Look-at menu (landing also clears the remembered pick); mode deactivation closes those plus the Observatory panel; event jumps close only the picker — the panel drives them and stays open by design. The panel itself sits outside the trio: it pairs one-instrument-at-a-time with the Stats card. The deck has a dim click-away backdrop, but it stacks UNDER the action cluster (backdrop 26 < deck 27 < cluster 28) so Leave and cluster tab-switching stay clickable while it's open — that's why the landing/takeoff close calls stay load-bearing; the Look-at menu (and the help modal) keeps a full-screen click-catcher. Wire new overlays into the same close calls. The tutorial card is the one deliberate exception: the tutorial stages scenes through the other overlays (the deck theater opens and closes under it at z 30), so no overlay's open() closes the card — only its own buttons and the tutorial lifecycle do.
- **UI idioms**: list rows use the deck's `pk-row` family (`pk-dot`, `pk-info`, `pk-tag-here`; the Look-at menu rides the same classes with `stm-*` overrides — its pill deliberately reads "current", never "here"); body tints come from catalog `color` (`planets/planetData`, `planets/moonData`), never hardcoded (the Sun, absent from the catalogs, is the one commented exception); body names render through `bodyDisplayName` — except deck rows, which deliberately show raw catalog names so search matches what the rows say; view sub-states (surface view, orbit details) are session-only. UI copy is plain and human — read the neighbouring strings first and match them.
- **The mobile breakpoint is 640px** (one small ≤420px block tweaks the Leave label and Stats card): test UI changes at desktop and at 390×844 — the narrow band is where overlays collide.
- Startup URL params: `?auto=planetarium` (and the legacy `?auto=moonView`, kept so old links boot the app) land in the Planetarium; `?debug=1` shows the error/debug overlay.
