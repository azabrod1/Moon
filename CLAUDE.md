## Architecture

A **Three.js** WebGL app: the **Planetarium** is the app, plus a dormant **Moon Flight** mini-game sharing the scene, renderer, and animation loop. `src/main.ts` is a thin entry point: it owns the renderer, the post-processing composer, the two cameras, mode switching/transitions, and the animation loop. Each mode is a self-contained controller that exposes `activate()/deactivate()/update(dt)`. (The legacy "Moon view" mode retired in favor of the Observatory â€” recoverable from git history; its orbit-details overlay was ported first.)

### Rendering Pipeline
- Post-processing bloom (`UnrealBloomPass`) on desktop when the GPU supports float framebuffers (tested at startup via a float FBO + `FRAMEBUFFER_COMPLETE`). Mobile/incapable GPUs get a no-bloom fallback with extra Sun glow shells. The composer is rebuilt when switching cameras between modes.
- PCF shadow maps for eclipse shadows. Custom GLSL: Sun corona (fbm noise), Earth night lights, and a per-planet single-scatter atmosphere shell (`shaders/atmosphere` â€” Rayleigh day-limb that warms toward the terminator + a Henyey-Greenstein Mie forward-scatter crescent, additive HDR so the composer's tonemap + bloom carry the bright rim).
- Per-body surface lighting is injected into the standard material through a shared `onBeforeCompile` hook (`world/surfaceShading`): a dim directional starlight night-floor, per-archetype limb darkening, Saturn's ring shadow, and moon-transit umbra spots â€” the GLSL is byte-identical per body (only uniforms differ) so programs still share. `world/planetshine` adds the earthshine night-side lift.
- Textures load through `world/texturePolicy`: device-capability capture, colour-space by map kind, and a capability-gated 2Kâ†’4K tier upgrade-on-approach (4K variants live under `public/textures/4k/` with identical filenames).

### Key Patterns
- **All distances are real**: the Planetarium works in AU throughout (1 scene unit = 1 AU).
- **All textures bundled locally** in `public/textures/` (not CDN), referenced via `import.meta.env.BASE_URL + 'textures/'`. Asset paths built from `BASE_URL` strings are invisible to both tsc and Vite â€” verify them by running the app.
- **UI is vanilla HTML** in `index.html` (no framework); each mode has its own overlay section toggled via `display`. The TS reads elements by `id` â€” keep the `index.html` id set and the `getElementById` calls in sync.
- **Nothing half-loaded is ever shown**: a moon never renders before `MoonPainter` finishes it, and teleports hold the arrival veil until the destination scene is ready. Perf work must preserve this gate.
- **Transient overlays are one-modal-at-a-time**: the â˜° menu, the deck, and the Look-at menu each close the others on open. Landing and takeoff close the deck and the Look-at menu (landing also clears the remembered pick); mode deactivation closes those plus the Observatory panel; event jumps close only the picker â€” the panel drives them and stays open by design. The panel itself sits outside the trio: it pairs one-instrument-at-a-time with the Stats card. The deck has a dim click-away backdrop, but it stacks UNDER the action cluster (backdrop 26 < deck 27 < cluster 28) so Leave and cluster tab-switching stay clickable while it's open â€” that's why the landing/takeoff close calls stay load-bearing; the Look-at menu (and the help modal) keeps a full-screen click-catcher. Wire new overlays into the same close calls. The tutorial card is the one deliberate exception: the tutorial stages scenes through the other overlays (the deck theater opens and closes under it at z 30), so no overlay's open() closes the card â€” only its own buttons and the tutorial lifecycle do.
- **UI idioms**: list rows use the deck's `pk-row` family (`pk-dot`, `pk-info`, `pk-tag-here`; the Look-at menu rides the same classes with `stm-*` overrides â€” its pill deliberately reads "current", never "here"); body tints come from catalog `color` (`planets/planetData`, `planets/moonData`), never hardcoded (the Sun, absent from the catalogs, is the one commented exception); body names render through `bodyDisplayName` â€” except deck rows, which deliberately show raw catalog names so search matches what the rows say; view sub-states (surface view, orbit details) are session-only. UI copy is plain and human â€” read the neighbouring strings first and match them.
